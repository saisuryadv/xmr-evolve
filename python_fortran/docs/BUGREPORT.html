MR3-GK Bidiagonal SVD — Comprehensive Bug Report

All defects discovered while making Willems' XMR kernel solve the bidiagonal SVD problem of Willems & Lang 2012 — both kernel patches and the orchestration-layer bugs surfaced during the Python and Fortran ports.

2026-04-30 · python_fortran/ · supersedes BUGREPORT.md (4 entries) and CHANGES.md (chronological).

K XMR kernel patch (Fortran) P Python orchestration bug H Py↔Fortran harmonization C Configuration / ablation

TL;DR

Two patches to Willems' XMR Fortran kernel and nine bug fixes in the Python orchestration were needed to take MR3-GK from 0/64 specs passing at the initial port to 379/379 specs at 100%. The kernel patches together unlock 190 specs (out of 379); the orchestration fixes account for the remaining ~150. Two more harmonization shims (reproducible RNG, unified BLAS) were needed to get bit-identity between the Python and Fortran orchestrations.

This document explains what each defect was, the root cause, the fix, and the empirical evidence that the fix matters. Bug IDs follow the convention used in the audit (K = kernel, P = Python, H = harmonization, C = config); chronology is given in the "From commit" column.

XMR kernel patches (linked into libxmr.so)

K1 — dlaxre_gk.f: GK-path was disabled kernel

Symptom

For any T_GK matrix (the perfect-shuffle of a bidiagonal B whose diagonal is identically zero — the matrix MR3-GK actually solves), the eigenvectors XMR returned were garbage. Singular values were correct (DSTEBZ computes them independently); orthogonality blew up to 10¹¹ × n·ε.

Root cause

xmr_src/dlaxre.f contains a branch dedicated to constant-diagonal tridiagonals (lines 269–340). It is guarded by IF(.FALSE.)THEN with a revealing comment: 

      IF( .FALSE. )THEN
C        Support for GK-type matrices deactivated.
C        Reason: One should call a specific SVD routine in this case,
*         see below. We ran into problems for GRO-type synthetics,
*         mainly because full blocks in the root restrict possible
*         choices for blocks in the children too much, due to
*         overlap sequences.

Willems disabled this branch in production because his test workload (general SYM/PD tridiagonals) was better served by Gershgorin shifts. But for bidiagonal SVD, T_GK is the input — it is exactly the case Willems steered around. Without the GK branch, T_GK falls through to a positive-definite shift that destroys the GK structure (||u|| = ||v|| = 1/√2) the algorithm relies on.

Fix

Replace the always-false guard with a real test: a constant-diagonal tridiagonal is one whose diagonal is small relative to the off-diagonals.

      IF( ABS(DMAX)+ABS(DMIN) .LT. EPS*EMAX )THEN
C        GK-type matrix: constant (zero) diagonal.
C        Use T_GK directly as root per Algorithm 4.1 (Willems-Lang 2012).
C        No outside shift needed — preserves GK structure.

Empirical evidence

Reverting only K1 (variant libxmr_origre.so) on the 379-spec suite:

  • 233 / 379 pass (146 failures) −146
  • Worst regressions (ortU column, patched values in parens): exponential_graded@10 1.32 × 10¹² (0.15); gl_gradp@100 5.89 × 10¹¹ (0.03); huge_condition@100 4.57 × 10¹¹ (0.28).

Reference

Willems & Lang, Computing the bidiagonal SVD using MR³, ETNA 39 (2012), Algorithm 4.1; LAWN 163 §5.4.

K2 — dlaxrb_clssfy_fix.f: MIN(0, x) collapse at depth 0 kernel

Symptom

On tight-cluster matrices (Demmel S1, three_clusters, pd_T0, gl_abcon, gl_wilkw), the representation tree never created clusters — every adjacent eigenvalue pair was classified as a "full gap". Singletons multiplied; orthogonality degraded to 30–2800 × n·ε on the worst specs.

Root cause

The classifier in xmr_src/dlaxrb_clssfy.f (and twice more at lines 414, 445) decides whether a candidate gap is a real cluster boundary: 

      AVGTHRESH = TWO * AVGTOL          ! AVGTOL ∝ 2^(-DEPTH)
      GAPTHRESH = TWO * GAPTOL
      ...
      IF( SPREAD .GE. MIN( AVGTHRESH, ABSMAX*GAPTHRESH ) )THEN
         RGINFO(J) = GI_FULLGAP
      ELSE
         RGINFO(J) = GI_NOFULL
      ENDIF

At depth 0, the parent has not yet computed an average gap, so AVGTOL = 0 → AVGTHRESH = 0 → MIN(0, ABSMAX·GAPTHRESH) = 0. The test becomes SPREAD ≥ 0 — trivially true. Every gap is a full gap, clusters never form, and the subsequent MR3 pipeline runs without the protection that clustering provides for tightly-spaced eigenvalues.

Fix

Case-split: if AVGTHRESH has underflowed to zero, fall back to the relative-only test. Three sites, each one wrapped in:

      LOGICAL  ::  GAPOK
      ...
      IF( AVGTHRESH .GT. ZERO )THEN
         GAPOK = SPREAD .GE. MIN( AVGTHRESH, ABSMAX*GAPTHRESH )
      ELSE
         GAPOK = SPREAD .GE. ABSMAX*GAPTHRESH
      ENDIF
      IF( GAPOK )THEN
         RGINFO(J) = GI_FULLGAP
      ELSE
         RGINFO(J) = GI_NOFULL
      ENDIF

Empirical evidence

Reverting only K2 (variant libxmr_origcls.so):

  • 330 / 379 pass (49 failures) −49
  • Worst regressions: ST_B_20_graded ortU 2782 (0.26); demmel_S1pe_k4@100 ortU 33.2 (3.62); three_clusters@100 ortU 21.6 (0.23).

K1 and K2 are synergistic: 13 specs pass under either single revert but fail under double revert (variant libxmr_origboth.so: 189/379).

Reference

Discovered while debugging score 86.37 → 92.39 transition (commits 6319178, 6f42a80, 2026-04-02). The 6f42a80 cleanup removed an earlier MAX(DEPTH, 1) hack in favor of the explicit conditional.

Python orchestration bugs (in mr3_gk.py)

These are bugs in our wrapper code — not in XMR. They were discovered while building the bidiagonal-SVD layer on top of XMR's tridiagonal eigenvector kernel. None of them existed in XMR; XMR doesn't do bidiagonal SVD at all, so it can't have these bugs.

P1 — Negative off-diagonal in T_GK breaks XMR's dlaxre orchestration

Symptom

When the bidiagonal B has any negative entry in d or e, XMR's root-representation builder produced wrong eigenvectors (residuals 0.86, not 1e-15). Singular values were fine.

Root cause

T_GK = perfect_shuffle(0, |d|, |e|) is the matrix MR3 actually solves. For B with sign pattern, T_GK gets signed off-diagonals at positions 2i. XMR's dlaxre assumes positive off-diagonals throughout (its qd-form factorization uses e directly as a square root). Negative e[i] silently builds a representation for a different matrix.

Fix

Apply a diagonal similarity transform D·T_GK·D with D = diag(±1) chosen so all off-diagonals are positive, then undo the similarity on the eigenvectors:

signs = np.ones(m2n)
for i in range(m2n - 1):
    if tgk_e[i] < 0: signs[i+1] = -signs[i]
    else:             signs[i+1] =  signs[i]
tgk_e_pos = np.abs(tgk_e)
# Call XMR on positive-offdiag matrix:
w, Z, info = xmr_eigenvectors(root, tgk_e_pos, …)
# Undo similarity on eigenvectors:
Z[:, j] *= signs

Reference

Phase 2 of the initial port (CHANGES.md). The diagonal-similarity trick is standard for sign-asymmetric tridiagonals.

P2 — T_GK eigenvectors are for |B|, not B; need D1, D2 sign matrices orchestration

Symptom

Even after P1, the U and V columns reconstructed from T_GK eigenvectors gave the wrong SVD: σ values matched, but B − UΣVᵀ was huge.

Root cause

T_GK is built from |d| and |e| (XMR needs positive entries). Its eigenvectors therefore correspond to |B|, the bidiagonal with all entries replaced by their absolute values. To recover singular vectors of the original B, we need diagonal sign matrices D1 (for U) and D2 (for V) such that B = D1 · |B| · D2.

Fix

d1 = np.empty(n); d2 = np.empty(n)
d1[0] = 1.0
d2[0] = np.sign(d[0])
for i in range(n - 1):
    d2[i+1] = np.sign(e[i])    / d1[i]
    d1[i+1] = np.sign(d[i+1])  / d2[i+1]
# After XMR returns eigenvector pairs (V_part, U_part):
V[:, j] = ev[0::2] * d2     # even rows of T_GK eigenvector → V column, scaled by d2
U[:, j] = ev[1::2] * d1     # odd rows → U column, scaled by d1

Reference

Initial port (Phase 2). Standard derivation of GK from B; see Lawn 163 §5.

P3 — Bidiagonal e signed even after P1; T_GK builder also needs abs(e) orchestration

Symptom

Latent bug behind P1: even though the call into XMR was abs-cleaned, the construction of the T_GK matrix that we feed into XMR was using the raw signed e. The two paths disagreed and the result was corrupted entries 2i of T_GK.

Root cause

Single-line oversight in mr3_tgk_multiblock: 

# wrong:
be = e_bidiag[bbeg:bend].copy()
# correct:
be = np.abs(e_bidiag[bbeg:bend])

Fix

Take abs at construction time. The sign is recovered later via D1/D2 (P2).

Reference

Commit 30803e2, 2026-03-25 ("Fix T_GK construction for negative e entries"). One-line patch but the symptom looked identical to P1 and was caught separately.

P4 — T_GK with zero off-diagonal is reducible; XMR returns garbage on one half orchestration

Symptom

On any matrix with an exact zero in e (not from underflow — from the test generator: diagonal_only, certain glued matrices, etc.), one half of the output U or V was correct and the other half had ortU ~ 1.

Root cause

If bte[i] ≈ 0 for some i, the T_GK block is reducible: rows [start..i] and [i+1..end] are decoupled. XMR doesn't detect this — it builds a single representation across the boundary, and the eigenvectors mix contributions across two truly independent sub-problems. The sub-problem with the larger eigenvalues "wins"; the smaller-eigenvalue sub-block gets a degenerate eigenvector.

Fix

Detect zero positions and split T_GK into independent sub-blocks before calling XMR:

zero_positions = np.where(np.abs(bte) < SAFMIN)[0]
if len(zero_positions) == 0:
    w, Z = _solve_tgk_block(bc, bte, m2k, bk_hint=bk)   # standard path
    return w[:bk], Z[:m2k, :bk]
# else: split into [start..zp+1) sub-blocks, solve each independently
boundaries = []
start = 0
for zp in zero_positions:
    boundaries.append((start, zp + 1))
    start = zp + 1
if start < m2k:
    boundaries.append((start, m2k))
# solve each, assemble the global (w, Z)

Reference

Commit f623667, 2026-03-25. Standard "irreducibility" reasoning for unreduced tridiagonals.

P5 — Singleton blocks trigger O(n³) GS pathology orchestration

Symptom

chkbd@400 timed out (>60 s) at the GS-completion stage. Profiling showed >99% of time in bidiag_svd's orthogonal-completion loop.

Root cause

After bidiagonal splitting, many blocks have size 1 (singletons). For a singleton, the 2×2 T_GK is trivially diagonal: σ = |d|, eigenvector = e_i for B-row, e_j for the other side. Calling XMR on the trivial 2×2 wastes time but, worse, XMR returns vectors with ||v||=1 on one side and ||u||=0 on the other. bidiag_svd's completion loop then sees hundreds of columns with ||u|| < 0.5 and runs Gram-Schmidt for each one against the n basis vectors → O(n²) per column × n columns = O(n³). For n=400 that is 64 million flops, dominated by Python overhead.

Fix

Bypass XMR for singletons: σ = |d_i|, both U and V contributions = 1/√2 directly.

for bbeg, bend in blocks:
    if bend == bbeg:                           # singleton block
        sigma[col]   = abs(d[bbeg])
        U[:, col]    = e_unit(2*bbeg + 1)      # row 2k+1 of T_GK  → U
        V[:, col]    = e_unit(2*bbeg)          # row 2k   of T_GK  → V
        col += 1
        continue

Empirical evidence

chkbd@400: >60 s timeout → 0.025 s. Score 5.0 → 82.97.

Reference

Commit 4bcdee9, 2026-03-24 ("Singleton bypass optimization").

P6 — Bv recovery threshold scaled with σ instead of ||B|| orchestration

Symptom

On matrices with a wide singular-value spectrum (σ_min ≪ σ_max), the tiny columns of U/V developed huge orthogonality errors. The very largest σ were fine.

Root cause

After the MR3 solve, bidiag_svd checks B·v vs σ·u. If one side has worse residual it recomputes that side from the other. The threshold for "this side is bad" was 10·n·ε·σ_idx. For tiny σ this threshold is tiny, so the recovery fires, and the recovery formula u_new = (B·v)/σ divides by tiny σ, amplifying any residual roundoff catastrophically.

Fix

Threshold should be a property of the matrix, not the singular value:

# wrong (sigma-based):
thresh_arr = 10 * n * EPS * sig_idx
# correct (bnorm-based, never collapses):
bnorm_val  = max(np.max(np.abs(d)), np.max(np.abs(e[:n-1])) if n > 1 else 0.0)
thresh_arr = 10 * n * EPS * np.maximum(sig_idx, bnorm_val)

Reference

Commit 965fd82, 2026-03-24 ("Fix Bv recovery threshold").

P7 — GS completion: O(n³) candidate-vector search → two-pass MGS orchestration

Symptom

After P5 fixed the singleton case, larger blocks still suffered: when a few columns genuinely needed completion, the completion loop searched n candidate basis vectors per column, each search O(n²). Total: O(n³). Caused timing-budget failures even on matrices that were numerically correct.

Root cause

The original code, for each column j with ||v_j|| < 0.5, did: 

for j in needs_completion:
    v_cand = random_unit_vector(n)
    for k in range(n):                         # ← n basis vectors
        if k != j and np.linalg.norm(V[:,k]) > 0.5:
            v_cand -= np.dot(v_cand, V[:,k]) * V[:,k]    # ← n flops, in Python
    V[:, j] = v_cand / np.linalg.norm(v_cand)

Fix

Vectorize the projection as a single BLAS matmul (one pass), then do a second pass for stability — classical two-pass MGS:

good_mask = (all_v_norms > 0.5) & (np.arange(n) != j)
V_good = V[:, good_mask]                       # n × m matrix (m ≤ n)
v_cand -= V_good @ (V_good.T @ v_cand)         # ← one BLAS call, O(n·m) total
v_cand -= V_good @ (V_good.T @ v_cand)         # second pass for stability
V[:, j] = v_cand / np.linalg.norm(v_cand)

Empirical evidence

worst_ratio (timing-bound metric): 5.97 → 4.30. Score 5.0 → 85.77 (combined with P5 and P6).

Reference

Commit ab5b004, 2026-04-02. Classical Gram-Schmidt with re-orthogonalization (Björck, Numerics of GS Orthogonalization, 1994) — one of the few cases where "do it twice" really does help.

P8 — Zero singular value from product underflow can't be deflated by XMR orchestration

Symptom

Some specs (e.g. matrices where σ_min underflows due to a long product chain d_1·d_2·…·d_n) had σ exactly 0 in IEEE-754 even though no individual e[i] was zero. XMR's classifier doesn't deflate this case (it keys on e, not on σ); the all-zero singular value is treated as a regular singleton, leading to ortU ~ n.

Root cause

Floating-point underflow in the σ product. Cannot be detected from looking at d, e alone — only revealed when σ is computed.

Fix (Demmel-Kahan 1990 zero-shift QR sweep)

A zero-shift QR sweep on an upper bidiagonal is mathematically equivalent to inverse iteration on B BT; if σ_min = 0, one sweep drives both d[-1] AND e[-1] to zero (numerically, below n·ε after prescaling). This gives a clean split: solve the (n−1)-subproblem, embed the σ=0 column at index n. 

def zero_shift_qr_sweep(d, e):
    """One downward sweep of implicit zero-shift QR. Modifies d,e in place.
       Demmel-Kahan 1990, Sec 3."""
    n = len(d)
    if n <= 1: return [], []
    cs, oldcs, oldsn = 1.0, 1.0, 0.0
    right_rots, left_rots = [], []
    for i in range(n - 1):
        cs, sn, r = dlartg(d[i] * cs, e[i])
        right_rots.append((cs, sn, i))
        if i > 0: e[i-1] = oldsn * r
        oldcs, oldsn, d[i] = dlartg(oldcs * r, d[i+1] * sn)
        left_rots.append((oldcs, oldsn, i))
    h = d[n-1] * cs
    d[n-1] = h * oldcs
    e[n-2] = h * oldsn
    return right_rots, left_rots
# In bidiag_svd:
d_sw, e_sw = d[bbeg:bend+1].copy(), e[bbeg:bend].copy()
right_rots, left_rots = zero_shift_qr_sweep(d_sw, e_sw)
clean_split = abs(d_sw[-1]) < n*EPS  AND  abs(e_sw[-1]) < n*EPS
if clean_split:
    sigma_sub, U_sub, V_sub, _ = bidiag_svd(d_sw[:-1] / scale, e_sw[:-1] / scale)
    # embed (bk-1) sub-solution into bk×bk, then apply Givens rotations

Why both d[-1] AND e[-1]

A zero-shift QR sweep transforms B → R Bᵀ Q where the bottom-right is exactly zero when σ_min = 0. Checking only d[-1] can misclassify near-singular cases where the sweep moved energy into e[-1] instead. Demmel-Kahan §3 guarantees both vanish only when σ_min is exactly representable as zero.

Reference

Commits 37412b8 (zero-shift sweep) and d1e8109 (refined check, 2026-04-02). Demmel & Kahan, Accurate Singular Values of Bidiagonal Matrices, SIAM J. Sci. Stat. Comput. 11 (1990), §3.

P9 — Adaptive GAPTOL: small blocks need stricter clustering orchestration

Symptom

two_clusters@10 failed even with K1, K2, P1–P8 in place. ortU sat at ~10 for n_nonneg=10 specs. GS-completion couldn't recover because the underlying MR3 output was bad to begin with.

Root cause

MR3's singleton orthogonality bound is roughly C · m · ε / relgap, where relgap is the relative gap between adjacent eigenvalues. The default GAPTOL = 1e-3 works for moderate n but is too loose for n < 20: when the suite is small, the gap test fires too often, MR3 emits singletons too eagerly, and the m·ε/relgap term blows up because relgap is small for tightly-spaced σ in small blocks.

Fix

Scale GAPTOL with 1/n_nonneg so small blocks force tighter clustering:

# for each block:
block_gaptol = max(GAPTOL, 0.02 / n_nonneg)
# then call xmr_eigenvectors with gaptol=block_gaptol

0.02 / 20 = 1e-3 = GAPTOL — converges to the standard threshold at n=20 and gets stricter from there.

Reference

Commits a23d535 (initial 5e-3 floor for n≤20) and 9fb2235 (smooth 0.02/n), 2026-04-02. Pushed final score to 379/379.

Cross-implementation harmonization (Py↔Fortran bit-identity)

The orchestration logic was ported from Python (mr3_gk.py, ~640 lines) to Fortran (mr3gk_fortran/*.f90, ~1,400 lines). The goal was that running the two on the same input produces bit-identical σ and ≤1 ULP differences in U/V across 21,375 specs at 2·ε. Two harmonization shims were needed:

H1 — Reproducible RNG via LAPACK DLARNV harmonization

Symptom

~6 / 379 specs differed by 1–2 ULP between Py and Fortran orchestrations on σ=0 specs (where GS completion fills in random vectors).

Root cause

np.random.randn uses Mersenne Twister; even with a fixed seed, the bytes that come out are different from LAPACK's DLARNV(IDIST=3). So the Python side fed completion with one set of random vectors, the Fortran side with a different set, and the resulting orthonormal completions differed at the last bit.

Fix

Bind LAPACK DLARNV via ctypes on the Python side; use the same deterministic 4-tuple seed (derived from column index) on both sides:

_lapack_lib = _ct.CDLL(_ctutil.find_library('lapack'))
def _dlarnv_normal(n, seed):
    """N(0,1) samples via LAPACK DLARNV (IDIST=3).
       seed: 4 ints, each 0..4095, last must be odd."""
    iseed = (_ct.c_int * 4)(*seed)
    n_c, idist = _ct.c_int(n), _ct.c_int(3)
    x = np.zeros(n, dtype=np.float64)
    _lapack_lib.dlarnv_(_ct.byref(idist), iseed, _ct.byref(n_c),
                        x.ctypes.data_as(_ct.POINTER(_ct.c_double)))
    return x
# At call sites:
v_random = _dlarnv_normal(n, (1, 3, 5, (2*j + 1) & 4095 | 1))    # V completion
u_random = _dlarnv_normal(n, (2, 4, 6, (2*j + 3) & 4095 | 1))    # U completion

The Fortran side calls DLARNV(3, iseed, n, …) with the identical seed sequence (mr3gk_postproc.f90:371-377, 439-443).

Empirical evidence

Pre-fix: 6 ULP-level mismatches. Post-fix: bit-identical or ≤1 ULP across 21,375 specs.

H2 — Unified DNRM2 via system BLAS harmonization

Symptom

Even after H1, a handful of gl_wilkw/gl_wilk2w specs differed by ~1 ULP between Py and Fortran.

Root cause

np.linalg.norm calls numpy's bundled OpenBLAS. The Fortran side calls system reference BLAS (whatever -lblas links). The two implementations differ at the last bit on tricky inputs (different summation orders, different fused-multiply-adds). Every normalization in bidiag_svd inherited that 1-ULP delta.

Fix

Bind the same BLAS the Fortran side uses, via ctypes, and route every Python normalization through it:

_blas_lib = _ct.CDLL(_ctutil.find_library('blas') or _ctutil.find_library('lapack'))
_blas_lib.dnrm2_.restype  = _ct.c_double
_blas_lib.dnrm2_.argtypes = [_ct.POINTER(_ct.c_int),
                              _ct.POINTER(_ct.c_double),
                              _ct.POINTER(_ct.c_int)]

def _system_dnrm2(x):
    x_arr = np.ascontiguousarray(x, dtype=np.float64)
    n_c, inc = _ct.c_int(x_arr.size), _ct.c_int(1)
    return _blas_lib.dnrm2_(_ct.byref(n_c),
                            x_arr.ctypes.data_as(_ct.POINTER(_ct.c_double)),
                            _ct.byref(inc))
# Plus a column-wise variant _system_dnrm2_axis0 for matrix normalization.

Used at 10 sites in bidiag_svd's post-processing (mr3_gk.py:865-984).

Empirical evidence

Combined with H1: 379/379 of the bit-match test pass at 2·ε. Worst U diff = 1.110 × 10⁻¹⁶ (1 ULP), worst V diff = 2.272 × 10⁻¹⁹.

Configuration knob worth knowing about

C1 — USEBLOCKS = .TRUE.: 2×2 block factorizations configuration

What it is

xmr_src/dlaxrs_stat.f:68 exposes a compile-time switch LOGICAL, PARAMETER :: USEBLOCKS = .TRUE.. When true, XMR uses the generalized LDL* twisted factorizations of Willems-Lang 2011 (ETNA 38), in which D may contain 2×2 blocks. During the shift T_GK − τI = L·D·L*, whenever a pivot would become dangerously small (signaling element growth), a 2×2 block absorbs the growth. The criterion: |d_plus(i) · c_plus| < (1/8) · gnsq(i).

Why it matters

The 2012 paper's Example 4.8 shows that shifting a GK matrix by α ≪ 1 produces element growth ~1/α² in the standard LDL*. For B = [[1, 1], [0, α]], shifting by −α gives D(2) = (1 − α²) / (α · (2 − α²)) ≈ 1/α — destroys the relative-robust-representation property MR3 depends on. 2×2 blocks act as numerical shock absorbers: they store d⁺ᵢ·d⁺ᵢ₊₁ as a single 2×2 determinant instead of two separate pivots, avoiding the cancellation.

Status

USEBLOCKS = .TRUE. is the upstream default and we keep it. We did not modify this — but it's worth documenting because turning it OFF reveals which specs depend on it.
